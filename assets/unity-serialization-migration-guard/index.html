Why teams use it

Serialized-field migrations can break long after the rename.

A broken serialized-field migration can be difficult to spot until scenes, prefabs, or saved assets are opened later. This tool gives you evidence before cleanup.

  • You renamed a serialized field and do not fully trust that every asset updated.
  • You are adding FormerlySerializedAs and want to catch anything you missed.
  • You want a real coverage report, not guesswork.
  • You need a repeatable migration workflow for teams and CI.

Start with safe mappings

Define the rename plan before touching assets.

  • Review missing types, ambiguous type names, and missing target fields.
  • Catch non-serializable target fields before they silently fail to load old data.
  • See when the new field is missing the required FormerlySerializedAs attribute.
  • Keep a reusable mapping file for local work, review, and CI.

Use your git diff

Import rename candidates from real refactor changes.

  • Preview candidates before writing them into your mapping list.
  • Append or replace mappings based on the state of the refactor.
  • Keep audit and coverage tied to concrete code changes instead of memory.

Coverage scan

Find exactly where legacy field tokens still exist.

  • Counts remaining tokens per field and per asset.
  • Tracks unreadable assets so incomplete coverage is visible.
  • Supports scenes, prefabs, and .asset ScriptableObject assets, including parent assets with embedded ScriptableObjects.
  • Shows unsupported inputs instead of hiding them.

Guided reserialize

Preview the scope, then reserialize only the assets that matter.

  • Build a preview from coverage results, a folder, or the current selection.
  • See skipped inputs and warnings before the run starts.
  • Create optional backups for assets and their .meta files.
  • Track progress and cancel when needed.

Cleanup proof

Know when it is safe to remove the legacy mapping path.

  • Flags remaining legacy token occurrences.
  • Blocks unsafe cleanup when coverage is incomplete.
  • Exports JSON and CSV proof artifacts for review and CI logs.

Who it is for

Built for teams shipping refactors with serialized state.

  • Projects with scenes, prefabs, and ScriptableObject assets that carry long-lived data.
  • Teams that want a repeatable migration workflow instead of ad hoc cleanup.
  • CI pipelines that should fail when risky migration state still exists.

Who it is not for

Not meant for runtime-only changes or projects that do not need migration proof.

  • Runtime gameplay tuning that does not change serialized field names.
  • Projects that cannot or will not use Force Text for coverage and cleanup checks.
  • One-off prototypes where serialized migration safety is not a real risk.

FAQ

Common questions before rollout.

Does this require Force Text serialization?

Coverage Scan and Cleanup Proof require Project Settings > Editor > Asset Serialization Mode = Force Text. Audit does not.

What asset types does it support?

Scenes, prefabs, and .asset ScriptableObject assets, including parent assets with embedded ScriptableObjects.

Can it be used in CI?

Yes. It includes batchmode audit, coverage, and cleanup commands with stable exit codes.

Does it write assets immediately?

No. Audit and coverage are analysis steps. Reserialize uses a preview-first workflow and can optionally create backups before writing.
