Fast Play Mode Safety Guard documentation

Fast Play Mode Safety Guard is a Unity Editor utility for teams that want faster Play Mode entry with Domain Reload disabled, but still need to see static-state risks before they trust the workflow.

The tool reports evidence. It does not prove every gameplay path is safe, and it does not automatically rewrite user scripts.

1. Requirements

  • The project must compile so the tool can inspect compiled assemblies.
  • The scanner works from Unity Editor code and does not ship in the player build.
  • The Double Play test requires Enter Play Mode Options enabled with Domain Reload disabled.
  • By default, scans focus on user project assemblies. Package, Unity, editor-only, this tool's own assemblies, and bundled demo findings are hidden unless enabled in Settings.

2. Quick start

  1. Open Tools > Kevin Bjorvand > Fast Play Mode Safety Guard > Open Window.
  2. Click Scan Project.
  3. Review the summary, readiness status, scan scope, scan duration, and top active risk groups.
  4. Select a finding to read the risk explanation, reset evidence, and suggested fix.
  5. Fix the code, generate a reset stub, ignore a false positive, or mark a reviewed risk as handled.
  6. Export the JSON report when you need a snapshot for review or CI.
  7. With Domain Reload disabled, run the Double Play test for runtime persistence evidence.
  8. Use Enter Safe Fast Play after scanning when you want a guarded Play Mode entry point.

3. Main window

Open the window from:

Tools > Kevin Bjorvand > Fast Play Mode Safety Guard > Open Window

The window includes the scan summary, readiness status, active risk groups, filters, search, paging, a sortable table, and a detail panel for the selected risk.

  • Scan Project refreshes the current report.
  • Generate Reset Stub creates or updates the separate generated reset file.
  • Export Report writes a JSON report.
  • Run Double Play Test checks runtime persistence for scanned active risks.
  • Enter Safe Fast Play enters Play Mode directly when the report is safe enough, or warns when active risks remain.

4. Readiness status

StatusMeaning
UnsafeAt least one active critical risk is present.
NeedsReviewAt least one active high-risk item is present.
ProbablySafeWarnings remain, but no active high or critical risks were found.
SafeEnoughForFastPlayModeNo active warning, high, or critical risks remain.

Ignored, manually handled, and covered findings stay visible as evidence, but they do not count as active risks.

5. Risk categories

  • StaticEvent and StaticDelegate: critical because old subscribers can survive later Play Mode sessions.
  • StaticCollection, StaticDictionary, and StaticArray: high risk because runtime data can accumulate or stay changed.
  • StaticUnityObjectReference: high risk because cached scene or asset objects can become stale.
  • SingletonInstance: high risk because Instance fields can keep old state alive.
  • InitializationFlag: high risk because flags such as IsInitialized can skip setup on the next run.
  • StaticServiceLocator: high risk because global service caches often survive when Domain Reload is disabled.
  • MutableStaticField: warning for other mutable static fields that may still need reset coverage.
  • RuntimePersistence: runtime evidence produced by the Double Play test.

6. Reset coverage

StatusMeaning
NoneNo reset coverage was found.
PotentialA reset-like method exists, but exact safe member coverage was not proven.
CoveredThe tool found explicit safe evidence, such as a safe attribute, handled setting, generated reset coverage, or verified reset assignment.
IgnoredThe member is suppressed in project settings or with an ignore attribute.
ManuallyHandledThe member is reviewed and handled outside what the scanner can prove.

Exact reset coverage is conservative. References, delegates, and events must reset to null; initialization flags must reset to false; value fields are covered only when compiled IL proves a zero or default assignment.

7. Attributes

Use attributes only after a member or type has been reviewed.

[FastPlayModeSafe]
[FastPlayModeIgnore("Accepted false positive.")]
[FastPlayModeResetHandled("Reset by project bootstrap.")]
  • Use FastPlayModeIgnore for accepted false positives.
  • Use FastPlayModeResetHandled when the risk is real but the lifecycle handling exists outside what the scanner can prove.
  • Use FastPlayModeSafe only when the static state is intentionally safe for Fast Play Mode.

8. Generated reset stub

Generate Reset Stub creates a separate file:

Assets/FastPlayModeSafetyGuard/Generated/FastPlayModeGeneratedReset.cs

The generator does not edit user scripts. It emits compile-safe reset assignments only when the field, declaring type, compile access, and reset value are proven safe.

Unsupported members, such as events, private fields, readonly fields, generic types, inaccessible assemblies, or unsupported value types, are written as comments. Review generated code before relying on it in production.

Preferred reset pattern:

using UnityEngine;

public sealed class GameManager
{
    public static GameManager Instance;

    [RuntimeInitializeOnLoadMethod(RuntimeInitializeLoadType.SubsystemRegistration)]
    private static void ResetStaticState()
    {
        Instance = null;
    }
}

9. Double Play test

Run Double Play Test is a runtime check for scanned active risks. It requires Enter Play Mode Options with Domain Reload disabled.

  1. Capture a static-state snapshot.
  2. Enter and exit Play Mode.
  3. Capture a second snapshot.
  4. Enter and exit Play Mode again.
  5. Capture a final snapshot and report persistence evidence.

The test can flag retained subscribers, initialization flags that stay true, collections that keep or grow items, singleton references that remain non-null, and scalar values that changed and stayed changed.

10. CI usage

Run the scanner in Unity batch mode:

Unity.exe -batchmode -quit -projectPath "ProjectPath" -executeMethod FastPlayModeSafetyGuard.CI.RunScan

Override the report path per run:

Unity.exe -batchmode -quit -projectPath "ProjectPath" -executeMethod FastPlayModeSafetyGuard.CI.RunScan -fastPlayModeReportPath "Library/FastPlayModeSafetyGuard/CIReport.json"
Exit codeMeaning
0The configured threshold passed.
1The configured risk threshold failed.
2Editor readiness blocked the scan.

CI uses the saved settings asset, exports JSON, logs a summary, and can fail on warnings, high risks, or critical risks based on settings.

11. Settings

The settings asset is stored at:

Assets/FastPlayModeSafetyGuard/Settings/FastPlayModeGuardSettings.asset
  • Include or exclude package assemblies.
  • Include or exclude Unity assemblies.
  • Include or exclude editor-only assemblies.
  • Include or exclude this tool's own assemblies and bundled demo findings.
  • Configure CI failure behavior and severity threshold.
  • Configure the default JSON report path.
  • Review and remove ignored or manually handled entries.

12. Limitations

  • The scanner works from compiled assemblies. Comment-based suppression is not implemented.
  • Reset coverage is Covered only when there is explicit safe evidence.
  • Generated reset stubs are conservative, created on demand, and do not modify user scripts.
  • The Double Play test only inspects static risks already found by the scanner.
  • The Double Play test provides runtime persistence evidence, not full gameplay coverage.
  • Unusual generated assemblies may need settings changes or future classifier tuning.
  • A project with compile errors may prevent the tool from loading.

13. Demo, license, and support
