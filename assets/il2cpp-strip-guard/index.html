Why teams use it

IL2CPP failures are often found late, when build-only debugging is already expensive.

Projects that rely on reflection, runtime type loading, custom serializers, save systems, registries, dependency injection, Addressables, generic factories, or modding-style systems can hide risks until a platform build is created.

  • String-based type lookup can be stripped if preserve coverage is missing.
  • Reflection member access can fail when metadata is unavailable.
  • Dynamic generic construction can expose AOT coverage gaps.
  • Serialized type names can point at runtime construction paths that need review.

IL2CPP Strip Guard turns those patterns into visible findings before release pressure takes over.

Project scan

Scan C# source and serialized assets for common IL2CPP risk patterns.

The scanner reviews C# files, serialized assets, existing Assets link.xml files, generated preserve files, and Preserve attributes in the configured scope.

  • Detects Type.GetType, assembly.GetType, Activator.CreateInstance, reflected member lookup, Invoke, MakeGenericType, MakeGenericMethod, open generic metadata, and SerializeReference signals.
  • Flags managedReferenceFullTypename, managedReferenceFieldTypename, and serialized strings that look like type references.
  • Supports package scanning, generated-folder scanning, Addressables scanning, and ignore rules through settings.

Preserve coverage

Separate active findings from risks already covered by preserve sources.

Findings remain visible for auditability, but recognized preserve coverage changes their status so review can focus on active risks.

  • Recognizes UnityEngine.Scripting.Preserve, Preserve, assembly preserve attributes, and Assets link.xml files.
  • Handles assembly-level, namespace-level, type-level, and member-level coverage where applicable.
  • Does not count package-local link.xml files as Unity-applied project coverage.

Generated fixes

Preview generated link.xml entries before writing suggestions.

The generator writes valid XML under the tool's generated folder, groups entries by assembly, avoids duplicates, and refuses unsafe output paths.

  • Default path: Assets/IL2CPPStripGuard/Generated/link.xml.
  • Uses preserve="all" for generated type entries with concrete type names.
  • Does not overwrite existing user link.xml files or modify source code automatically.
  • Optional AOT hint generation is disabled by default and stays conservative.

CI reports

Export JSON and Markdown reports, then fail CI by configured severity thresholds.

CI mode loads saved settings, runs the scanner without opening the editor window, builds a generated link.xml preview, exports reports, logs counts, and exits based on thresholds.

  • Default report folder: Library/IL2CPPStripGuard/Reports.
  • Default CI thresholds fail on active Critical and High findings.
  • Use IL2CPPStripGuard.Editor.IL2CPPStripGuardCI.Run or the fully namespaced equivalent.

Who it is for

Built for Unity developers shipping IL2CPP targets.

  • WebGL, Android, iOS, VR, and console-like projects.
  • Projects using reflection, runtime type loading, or custom serializers.
  • Save systems, command registries, event systems, dialogue systems, action systems, dependency injection, Addressables, and generic factories.

What it does not claim

Not a mathematical proof that every IL2CPP build is safe.

  • The scanner is heuristic and false positives or false negatives are possible.
  • It does not run player builds or validate platform builds.
  • Generated link.xml preserves metadata, but it may not force every closed generic instantiation.

FAQ

Common questions before rollout.

Does it modify my source code?

No. It never modifies user source code, player settings, existing user link.xml files, or build settings automatically.

Can it generate link.xml suggestions?

Yes. You can preview generated link.xml output and write it only when you choose to generate the file.

Can it run in CI?

Yes. The CI runner exports JSON and Markdown reports and exits with code 1 when configured thresholds are exceeded.

Does it prove generic AOT coverage?

No. link.xml can preserve metadata, but dynamic generic construction may still need explicit runtime references or hand-authored AOT hints.
