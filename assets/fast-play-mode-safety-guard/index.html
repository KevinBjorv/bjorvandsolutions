Why teams use it

Disabling Domain Reload is fast, but stale static state is hard to spot by hand.

Unity can keep static fields alive between Play Mode sessions. That helps iteration speed, but it can also preserve old subscribers, singleton instances, initialization flags, collections, and cached scene references.

  • Play Mode behaves differently on the second run than the first.
  • Tests pass or fail depending on previous editor state.
  • Event buses retain subscribers that should have been cleared.
  • Singletons and service locators keep old references alive.

Fast Play Mode Safety Guard turns that risk into a report your team can review and fix.

Static-state scan

Find the risky static members before they leak into Play Mode.

The scanner reviews compiled project assemblies and reports the static patterns most likely to survive when Domain Reload is disabled.

  • Static events and delegates that can retain old subscribers.
  • Static collections, dictionaries, arrays, and mutable fields.
  • Singleton instances, service locators, and initialization flags.
  • Cached UnityEngine.Object references that can become stale.

Readiness status

Know whether the report is unsafe, needs review, probably safe, or safe enough.

Each scan rolls findings into a project-level status so the next action is clear without hiding the details.

  • Unsafe when active critical risks remain.
  • NeedsReview when high-risk items remain.
  • ProbablySafe when warnings remain but no high or critical risks are active.
  • SafeEnoughForFastPlayMode when no active warning, high, or critical risks remain.

Reset coverage

Separate real fixes from accepted risks and false positives.

Every finding includes reset coverage evidence, suggested handling, and a clear distinction between active, ignored, manually handled, and covered items.

  • Use attributes such as [FastPlayModeSafe], [FastPlayModeIgnore], and [FastPlayModeResetHandled].
  • Generate a separate reset stub instead of rewriting user scripts.
  • Keep unsupported generated assignments as comments for review.
  • Prefer RuntimeInitializeOnLoadMethod with SubsystemRegistration for reset code.

Runtime evidence

Run Play Mode twice and check whether scanned risks persist.

The Double Play test captures static-state snapshots across two Play Mode passes and reports persistence evidence for scanned active risks.

  • Requires Enter Play Mode Options with Domain Reload disabled.
  • Flags retained event subscribers, collections, singletons, flags, and changed scalar values.
  • Gives focused runtime evidence without claiming full gameplay coverage.

CI reports

Export JSON evidence and fail builds by severity threshold.

Use the batchmode entry point when Fast Play Mode readiness needs to become part of a repeatable team workflow.

  • Run FastPlayModeSafetyGuard.CI.RunScan from Unity batch mode.
  • Export a JSON report with scan-settings and generated-reset fingerprints.
  • Configure CI failure behavior by warning, high-risk, or critical thresholds.

Who it is for

Built for teams that want faster iteration with visible state risk.

  • Unity projects considering Domain Reload disabled.
  • Teams with singleton, service locator, event bus, or static manager patterns.
  • Projects with tests that need repeatable editor state.
  • CI workflows that should block unsafe Fast Play Mode rollout.

Who it is not for

Not meant to prove every runtime path is bug-free.

  • Projects that keep standard Domain Reload enabled and do not need the audit.
  • Teams expecting automatic fixes inside user scripts.
  • Gameplay test coverage, since Double Play only checks scanned static risks.

FAQ

Common questions before rollout.

Does it edit my scripts?

No. The reset generator writes a separate generated file and comments unsupported fields instead of modifying user code.

Does it prove the whole project is safe?

No. It finds common static-state risks and gives reset evidence. Double Play provides runtime evidence for scanned static members, not full gameplay coverage.

Why are package and Unity assemblies hidden by default?

The first report should focus on code the project owner can usually fix directly. Broad scopes can be enabled in Settings when needed.

Can it run in CI?

Yes. It includes a Unity batchmode entry point that exports JSON and returns stable exit codes for configured thresholds.
