Mobile SDK Conflict Doctor Documentation

Mobile SDK Conflict Doctor is a Unity Editor preflight for diagnosing Android SDK integration conflicts.

It inventories AAR, JAR, androidlib, Maven, POM, EDM4U, manifest, Gradle, properties, rules, and native-library evidence while preserving the owning package, source path, archive entry, and XML line.

The Doctor explains project evidence and imports supported build-log signals. It does not replace EDM4U, Gradle, Android Studio, SDK installers, or vendor compatibility guidance.

1. Requirements

  • Unity 2022.3.41f1 or newer.
  • Android Build Support for real Android builds; source-only scans do not require it.
  • An Android-targeting Unity project with SDK, dependency, manifest, archive, or build-log evidence to review.

Release validation covered Unity 2022.3.41f1, 6000.3.9f1, and 6000.4.4f1. The real Android build was exercised on 2022.3.41f1 with Gradle 7.5.1, Android Gradle Plugin 7.4.2, and Java 11.0.14.1.

2. Installation

  1. Copy the complete Mobile SDK Conflict Doctor folder into Assets.
  2. Preserve every supplied folder and .meta file.
  3. Allow Unity to compile the Editor assemblies.
  4. Open Tools > Mobile SDK Conflict Doctor.

The implementation uses public Unity APIs and does not download SDKs or change Unity packages.

3. Quick Start

  1. Select Scan.
  2. Review the Detected, Likely, and Informational groups.
  3. Expand findings and open the owning files or source lines.
  4. Import a Gradle or Unity log after a failed build.
  5. Preview any supported remediation before confirming it.
  6. Export JSON, Markdown, HTML, or redacted support files.
  7. Save a baseline only after validating a healthy Android build.

4. Inventory and Evidence

Archives are read with safety bounds and never extracted. The report retains owners, archive entries, Maven states, source paths, and XML lines so a conflict can be traced back to durable project or package inputs.

  • AAR, JAR, androidlib, POM, Maven, and EDM4U declarations.
  • Android manifests, Gradle templates, properties, rules, and native libraries.
  • Requested, selected, forced, embedded, and excluded Maven states.
  • Unity, Gradle, AGP, Java, SDK, AndroidX, Jetifier, template, and ABI evidence when available.

Unknown values remain unknown. Source manifest analysis is a conservative prediction until final-build evidence confirms merged intent.

5. Findings and Durable Fixes

Detected means direct conflicting evidence. Likely means project evidence still needs build or vendor confirmation. Informational is relevant without claiming a conflict.

  • Duplicate Java classes and artifacts.
  • Incompatible Maven constraints.
  • Manifest provider or attribute conflicts.
  • Duplicate native libraries and missing ABIs.
  • AndroidX or Jetifier mismatches.
  • Obsolete Gradle workarounds and environment mismatches.

Prefer vendor-supported SDK combinations and edit durable declarations rather than generated Gradle output. The numerically newest dependency is not automatically compatible.

6. Failed Build and Log Workflow

  1. Run a fresh scan.
  2. Import the relevant Gradle or Unity build log.
  3. Check whether a versioned signature is marked build-confirmed.
  4. Navigate from the finding to the durable source.
  5. Export and manually review the redacted support set.

An unsupported error stays Unrecognized. The Doctor does not fabricate a diagnosis or rewrite the message into a guessed fix.

7. Baselines and Comparisons

Save Baseline writes ProjectSettings/MobileSdkConflictDoctor/Baseline.json. Save and commit a baseline only after an intentional, healthy Android build.

Comparison reports added, removed, and changed normalized nodes. Stable IDs and content hashes derive from normalized evidence, while scan IDs and timestamps intentionally vary.

8. Reports and Privacy

Exports include machine-readable JSON plus Markdown and standalone HTML review views. The support-redacted/ directory contains redacted JSON, Markdown, HTML, and a SHA-256 manifest.

Redaction covers common home-path and secret shapes but cannot guarantee removal of vendor-specific credentials. Review every file before sharing, and never include signing files or raw secrets.

9. Batch Mode and CI

Run the public batch entry point with:

-batchmode -nographics -projectPath <project> -executeMethod KevinBjorvand.MobileSdkConflictDoctor.MobileSdkConflictDoctorBatch.Run -mobileSdkDoctorOutput <directory>

Options include -mobileSdkDoctorThreshold detected|likely|information|none, -mobileSdkDoctorLog, -mobileSdkDoctorBaseline, and -mobileSdkDoctorIncludeGenerated.

  • 0: pass
  • 2: threshold met
  • 3: incomplete scan
  • 4: invalid input or export
  • 5: unhandled failure

10. Remediation and Limitations

The only automatic 1.0 operation can change one explicit AndroidX or Jetifier property from false to true in a project-local Assets/Plugins/Android/gradleTemplate.properties, and only when resolver evidence makes the change unambiguous.

Preview, confirmation, exact backup, atomic write, verification, rescan, and rollback checks guard that operation. Missing, duplicate, already-true, ambiguous, package-owned, and generated targets stay manual.

  • Android only; iOS and CocoaPods are excluded.
  • No downloader, package mutation, or resolver replacement.
  • Manifest analysis does not reproduce Gradle merger semantics.
  • Log recognition is versioned and finite.
  • Most fixes require manual vendor review.
  • No successful-build guarantee.

11. Troubleshooting and Support

The menu is missing

Preserve the complete folder and its .meta files, then resolve Unity Console compilation errors.

Expected inputs are missing

Confirm the Assets and Packages scan scope, then inspect safety-limit findings.

No final manifest conclusion is shown

Source analysis is predictive. Supply merged-manifest or final-build evidence before treating the result as build-confirmed.

Apply is disabled

No unambiguous supported durable edit exists. Follow the manual vendor-supported recommendation instead.

A scan is slow

Exclude generated output, use the conflict-only or collapsed graph view, or cancel through Unity's progress dialog.

Contact support, review the MIT license, or browse the Mobile SDK Conflict Doctor articles.
