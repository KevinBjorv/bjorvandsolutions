Package Quarantine Documentation

Package Quarantine makes a local .unitypackage import reviewable and recoverable.

It inspects the archive without extracting payloads into Assets, compares entries with the current project, explains risk evidence, creates a verified recovery transaction, opens Unity's native import dialog, and reconciles what actually changed.

Package Quarantine is evidence and recovery infrastructure. It is not an antivirus, execution sandbox, package downloader, UPM manager, dependency resolver, or replacement for version control.

1. Requirements

  • Unity 2022.3.41f1 or a validated newer Editor.
  • A local or cached .unitypackage file.
  • Enough free disk space for a stable project snapshot and byte backup under Assets, Packages, and ProjectSettings.

The Windows release suite passed 46 of 46 tests in Unity 2022.3.41f1, 6000.3.9f1, and 6000.4.4f1.

2. Installation

  1. Import the complete Package Quarantine asset into the project.
  2. Keep the supplied folders and .meta files together.
  3. Allow Unity to compile the Editor assemblies.
  4. Open Tools > Package Quarantine > Inspect & Import.

No external package or service is installed. The tool uses public Unity Editor and .NET APIs.

3. Quick Start

  1. Open Inspect & Import and choose or drag in a local .unitypackage.
  2. Wait for archive hashing and project comparison.
  3. Review classifications, collisions, code, settings, conflicts, and every expanded finding.
  4. Copy or export the Markdown dry-run report when approval is required.
  5. Acknowledge the findings and select Prepare Recovery & Import.
  6. Review the item selection in Unity's native import dialog.
  7. Open Recovery & History to keep the import or preview a rollback.

4. Inspection and Risk Evidence

Inspection reports archive paths, byte hashes, project conflicts, file types, package-control files, and recognizable risk signals without extracting payloads into the project.

  • Overwrite, code, settings, and conflict filters.
  • Managed and native plugin visibility.
  • Case and Unicode-equivalent collision detection.
  • Strict path containment and archive-format validation.
  • Package SHA-256 and visible evidence in the dry-run report.

A low score is not a safety verdict. Unknown code and binaries remain untrusted until independently reviewed.

5. Recovery Transactions

Recovery data is stored outside Assets at <project>/PackageQuarantineRecovery/. Each transaction includes journal generations, stable pre/post snapshots, verified backup blobs, import evidence, a staged package copy, rollback records, and manual restore instructions.

Before import, the tool takes consecutive stable snapshots and byte-verifies the original regular files in the managed project scope. The inspected package is staged and rehashed before Unity receives it.

6. Safe Rollback

Rollback compares the project before import, immediately after import, and now. Asset and .meta files are handled together so GUID identity can be restored.

  • Safe: current bytes still match the recorded post-import state.
  • Already desired: the path already matches the pre-import state.
  • Diverged: the path changed again and is left untouched by default.
  • Manual only: Package Quarantine will not automatically restore the operation.
  • Blocked: a containment, reparse-point, ownership, or safety check failed.

Rollback is restartable but is not one atomic filesystem transaction. Exact restoration is verified after Unity refreshes and compilation settles.

7. Reports

Before import, Copy Summary and Export Markdown Dry Run record classification and visible risk evidence. After reconciliation, transaction summaries can be exported as Markdown or standalone HTML.

Keep both reports when the pre-import review and the observed import outcome are required. The transaction report does not replace the detailed dry-run findings.

8. Compatibility

Package Quarantine is Editor-only and has no render-pipeline or optional package dependency. Windows was validated in Unity 2022.3.41f1, 6000.3.9f1, and 6000.4.4f1.

macOS and Linux recovery workflows have not been validated for 1.0.0. Do not infer host support because archive inspection uses portable .NET APIs; recovery also depends on host filesystem, link, permission, and path behavior.

9. Policy and Safety Limits

  • Package Quarantine does not prove that imported code is safe.
  • Editor code selected in Unity's dialog may run during import or refresh.
  • Recovery does not cover unsaved in-memory changes, VCS metadata, Library, Temp, Logs, UserSettings, or external side effects.
  • Packages/manifest.json and packages-lock.json are manual-only recovery targets.
  • Malformed or unsupported archive variants fail closed.
  • Version control remains recommended before third-party imports.

10. Troubleshooting and Support

A package is rejected before review

Confirm the file is a readable regular .unitypackage. Malformed gzip/tar data, traversal paths, unsupported extensions, links, and ambiguous paths fail closed.

Project bytes changed after inspection

The tool rehashes the source package and target evidence during preparation. Review the refreshed analysis before continuing.

Compilation failed after import

Open Tools > Package Quarantine > Recovery & History. The fallback window is isolated from user assemblies.

Manual recovery is required

Close Unity, copy the current project, and follow the transaction's RESTORE.md and restore-map.tsv. Restore assets and their .meta files together.

Contact support, review the license, or browse the Package Quarantine articles.
