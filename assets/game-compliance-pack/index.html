The problem

Indie and small studio projects routinely ship with a messy third-party dependency graph: Unity Package Manager dependencies (direct and transitive), Asset Store plugins and embedded third-party code, DLLs and native plugins, and small copied code folders. At release time you are forced to manually identify third-party components, track licenses, and build notices. It is error-prone and easy to forget until the last week.

What Game Compliance Pack does

  1. Scan your project for third-party components
  2. Resolve anything marked UNKNOWN with evidence shown
  3. Export deterministic, release-ready compliance files
It does not fabricate license IDs or license text. If something cannot be verified, it stays UNKNOWN until you resolve it.

What it scans

UPM

  • Direct dependencies from Packages/manifest.json
  • Resolved dependency graph (direct + transitive) from Packages/packages-lock.json
  • Captures package name, version, and source (registry, git, local path)

File system

  • Common plugin locations like Assets/Plugins and configurable third-party folders
  • Detects binaries and native plugins (.dll, .so, .dylib, .bundle)

How license resolution works (conservative by design)

  1. Manual override (your decision wins for that component key)
  2. Detected license or notice file near the component (uses extracted text as evidence)
  3. Known verifiable metadata mapped to canonical SPDX text stored locally
  4. UNKNOWN if nothing reliable exists

Built for stable, reviewable releases

  • Deterministic component keys and ordering so diffs are meaningful
  • Plain text outputs that are easy to audit
  • Optional build integration to warn or fail builds if UNKNOWN items remain

FAQ

Q: Does this guarantee I am compliant?

A: No. It is not legal advice and does not guarantee compliance. It reduces risk by making third-party items visible and forcing explicit resolution of unknowns.

Q: Will it detect everything in every project?

A: No. It targets the most common Unity dependency sources and keeps rules conservative. If something cannot be verified, it is marked UNKNOWN so you do not get false certainty.

Q: Does it upload my project or code anywhere?

A: No. It is local-first and runs inside the Unity Editor.

Q: What happens when something is UNKNOWN?

A: You resolve it once using the evidence shown (paths, hints, license files). The resolution is stored in a project file so builds remain reproducible across machines.

Q: What files do I ship?

A: Typically THIRD_PARTY_NOTICES.txt and CREDITS.md with your release. The manifest is your internal source of truth for audits and diffs.

Q: Is it only for UPM packages?

A: No. It also scans common third-party folders and plugin binaries.
